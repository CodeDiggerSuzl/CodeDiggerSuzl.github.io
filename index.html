正则的分组和反向引用

了解反向应用的前，首先了解正则中的分组，简单来说：组是用括号划分的正则表达式，可以根据组的编号来引用某个组。组号为 0 表示整个表达式匹配到的内容，组号 1 表示第一对括号扩起的组，以此类推，进行分组匹配时的原则是：由外向内，由左向右。

譬如一个手机号：16300020013，可以用(\d{3})\d{4}(\d{4})进行匹配。

这里整个手机号就是组 0，前三位是组 1，最后四位为组 2。

捕获组捕获到的内容，不仅可以在正则表达式外部通过程序进行引用，也可以在正则表达式内部进行引用，这种引用方式就是反向引用。

一种基于广播消息的 Redis 热key 的解决方法

发表于 2022-08-09 分类于 缓存 本文字数： 688 阅读时长 ≈ 1 分钟

背景

项目中最近遇到一个营销需求场景，类似于商品库存实时展示的功能。

简单来说就是每一个活动有一个奖池的概念。比如说总共有 999 个商品，用户做任务，完成任务就能领取奖池中奖励。

产品经理为了营造 “紧张的” 气氛，要求每秒轮询1次（先不谈是否有必要这么实时的查询）。那么当用户停留 H5 页面的时候，就会轮询去调用服务的接口，查询剩余奖品数。

QPS 大概 5-6 万左右。可能实际 QPS 没有这么高，只是一个大概的估计。

大致思路

计划是用一个 redis key 来维护这个数据，因为并不是秒杀场景，用户需要完成某个比较耗时的任务才能领取奖品。

可以通过 redis 的 incr 或者 decr 命令，在用户领取奖励时维护这个缓存。

维护这个 key 比较容易，但是读的频率要远远大于写频率。

本质上来说就是大量用户实时去请求 redis 中的维护的计数器。

redis 集群和其他业务组共用，希望尽可能的减少对 redis 的压力。防止对缓存成为系统的瓶颈。最坏的情况：reids 崩掉，那将对其他的业务组也造成极大的影响，因为一个查询的操作实在是得不偿失。

最终解法

我们最终采用了一种MQ 广播消息 + 本地缓存的思路。

  1. 当 redis 计数器的值变化的时候，用 RocketMQ 去发一个广播消息，然后将 redis 当前的值广播到各个应用实例上。
  2. 各个应用实例接收到这个广播消息的时候，将这个值维护在本地缓存中
  3. 请求来的时候只需要查询本地缓存就可以
  4. 活动时加机器，活动结束后摘掉实例。
  5. 加些兜底的策略
    1. 通过开关控制请求本地缓存还是 redis
    2. 出现异常返回假数据
    3. 等等符合业务需求的降级策略

这样避免了直接查询缓存可能带来的一些问题，算是一个比较巧妙的解决方法，故记录下。
